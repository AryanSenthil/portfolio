Abstract

This work presents the fabrication and characterization of a sustainable piezoelectric sensor utilizing agricultural waste in the form of onion peel as the active sensing material. The sensor was fabricated through direct ink writing (DIW) 3D printing, where dehydrated and ground onion peel powder was dispersed in a silicone matrix to create a functional composite ink. The self-aligned cellulose fibers in onion peel exhibit polarization under mechanical stress, enabling piezoelectric voltage generation. The sensor was tested under cyclic compression loading at three deformation levels (0.5, 0.7, and 1.0 mm) under two distinct loading configurations: point load and uniform load.

To extract actionable insights from the sensor's electrical response, we developed a novel two-stage machine learning architecture. The first stage employs convolutional neural networks (CNNs) on Short-Time Fourier Transform (STFT) spectrograms of the voltage-time signals to classify the type of applied load (point vs. uniform). The second stage uses a Wide & Deep regression model that combines the predicted load type with peak-to-peak voltage measurements to predict the magnitude of deformation. This architecture effectively captures the complex electromechanical behavior of the bio-based sensor, leveraging both memorization of load-specific patterns and generalization across the continuous deformation space. The results demonstrate the feasibility of transforming agricultural waste into functional sensing materials with intelligent signal processing capabilities for structural health monitoring and human-machine interface applications.

1. Piezoelectricity and Piezoelectric Effect

Piezoelectricity is the linear electromechanical coupling between mechanical and electrical domains.

Piezoelectric effect refers to an electric polarization induced by an applied mechanical stress ().

1.1 Mathematical Expression

$$ P_i = d_{ijk} \, \sigma_{jk} $$

where: $P_i$ = Electric polarization, $d_{ijk}$ = Piezoelectric coefficient tensor, $\sigma_{jk}$ = Applied mechanical stress

1.2 Why Piezoelectric Materials?

For piezoelectric materials, the crystal's structure is asymmetrical. When no external force is applied, the crystal is in a stable state and doesn't display electrical polarity to the outside. However, when an external force is applied, the crystal's atoms disperse and the internal charge distribution shifts, creating electrical polarity as the balance is disrupted.

Demonstration of piezoelectric effect: mechanical stress generates electrical charge

2. Fabrication of Onion Peel-Based Piezoelectric Sensor

2.1 Material Selection and Rationale

In this research, we explored onion peel as a sustainable piezoelectric material due to its self-aligned cellulose fibers that exhibit polarization under applied mechanical stress. This approach transforms agricultural waste into a functional sensing material.

2.2 Material Preparation Protocol

  1. Collection and Dehydration: Onion peel waste material was collected and thoroughly dehydrated
  2. Grinding: Dehydrated peels were ground using a high-speed grinder (3600 rpm)
  3. Sieving: Ground material was sieved using a vibrating sieve shaker to obtain uniform particle size distribution (roughly 25 $\,\mu\text{m}$)

2.3 Sensor Fabrication Process

Step 1: Composite Ink Preparation

  • Onion peel powder was dispersed in a silicone matrix (Ecoflex 00-30, 1:1 base:curing agent ratio)
  • The mixture was homogenized at 1500 rpm for 5 minutes to form an active composite ink

Step 2: 3D Printing via Direct Ink Writing (DIW)

  • The composite ink was printed layer-by-layer using a pneumatic DIW printer
  • Ink rheology was carefully tuned to ensure smooth extrusion and dimensional stability during printing

Step 3: Electrode Application

  • Conductive electrodes were applied to both surfaces of the printed sensor using silver paste to establish electrical contacts

2.4 Characterization and Testing

The fabricated sensor was connected to an electrometer to measure voltage generation under mechanical loading. Two loading conditions were evaluated:

  • Point Load: Localized force applied to a specific region of the sensor
  • Uniform Load: Distributed force applied across the entire sensor surface

As expected, voltage output was successfully measured under both loading conditions, confirming the piezoelectric response of the onion peel-based sensor.

4. Machine Learning Model Architecture for Load Characterization and Deformation Prediction

The piezoelectric response of our onion peel-based sensor exhibits distinct voltage-time signatures that correlate strongly with both the type of applied load (point vs. uniform) and the magnitude of deformation. We developed a novel two-stage machine learning architecture that leverages these correlations to simultaneously classify load type and predict deformation level from the sensor's electrical output.

4.1 Two-Stage Model Architecture

4.2 Load Classification Module

The cyclic mechanical loading produces characteristic voltage waveforms that vary distinctly between point and uniform load configurations. To classify these patterns, we treat the voltage-time data as audio signals and employ convolutional neural networks (CNNs) on their time-frequency representations.

4.2.1 Signal Processing Pipeline

Data Segmentation: Continuous voltage-time recordings are segmented into 10-second intervals, each saved as a WAV file

Time-Frequency Analysis: We apply the Short-Time Fourier Transform (STFT) to generate spectrograms

4.2.2 Mathematical Expression

$$ X(\tau, \omega) = \int_{-\infty}^{\infty} x(t) \cdot w(t - \tau) \cdot e^{-j\omega t} \, dt $$

where: $x(t)$ = voltage signal, $w(t-\tau)$ = window function (Hann window), $\tau$ = time frame, $\omega$ = angular frequency

STFT Visualization: Real-time generation of time-frequency spectrograms from cyclic voltage signals

4.2.3 Why STFT for Cyclic Loading Analysis?

Non-stationary Signal Characteristics: The voltage response during cyclic compression-decompression is inherently non-stationary, the frequency content changes as the sensor transitions between loading and unloading phases. STFT preserves temporal evolution, capturing how spectral components emerge and decay during each loading cycle.
Cycle-Specific Features: Point and uniform loads produce different stress distributions across the sensor, resulting in distinct harmonic patterns during compression. STFT reveals these time-localized frequency signatures. Point loads generate sharper transients with higher frequency components at the moment of contact, while uniform loads show more gradual spectral evolution.
Temporal Resolution: Our 10-second recording windows contain multiple compression cycles. STFT allows us to resolve individual cycles within this window, providing a detailed time-frequency map that the CNN can learn to associate with specific load types.
Robustness to Phase Variations: Since mechanical loading may not be perfectly synchronized across samples, STFT's sliding window approach provides translation-invariant features that are robust to slight timing variations in the compression cycles.

4.3 Deformation Prediction Module: Wide & Deep Regression

Initial experiments using only peak-to-peak voltage as a predictor for deformation showed promising results. However, incorporating the load type as an additional feature in traditional multi-layer perceptron architectures yielded minimal improvement, the categorical nature of the load type was being diluted during training.
We adapted the Wide & Deep architecture (Cheng et al., 2016), originally developed for recommender systems. This architecture elegantly combines memorization and generalization.

4.3.1 Architecture Components

Wide Component: A linear model that directly connects the load type (categorical feature) to the output, preserving its discriminative power through memorization

Deep Component: A deep neural network that processes the peak-to-peak voltage through multiple non-linear transformations, enabling generalization across the continuous deformation space

Wide & Deep Regression Architecture: Combining wide component (right) with deep component (left) for deformation prediction

This architecture leverages the complementary strengths of both features: the wide component maintains the strong correlation between load type and voltage response patterns, while the deep component captures the non-linear relationship between voltage amplitude and deformation magnitude.

4.4 Integration and Inference Pipeline

During inference, the system operates as follows:

1. Input voltage-time data is processed through the CNN classifier, outputting load type probabilities
2. The predicted load type (argmax of probabilities) is converted to binary representation
3. Peak-to-peak voltage is extracted from the raw signal
4. Both features feed into the Wide & Deep regression model for deformation prediction
Note on Feature Representation: While the CNN outputs a probability distribution over load types during inference, the regression model uses a discrete binary input (0 or 1) to maintain consistency with the training paradigm where ground truth labels were used.

4.5 Results

The integrated model achieved:

Load Classification Accuracy: 100%
Deformation Prediction RMSE: 0.0409 mm
These results demonstrate that the synergistic combination of spectral analysis for load classification and Wide & Deep regression for deformation prediction effectively captures the complex electromechanical behavior of our bio-based piezoelectric sensor.

Design Philosophy: Agent-Tool Architecture
Important Note for Code Reviewers and Lab Members:


The functions in this notebook are intentionally designed not to return models, plots, or other objects as variables. While this design choice may initially appear unconventional, it serves a deliberate architectural purpose aimed at maintaining modularity and system-level consistency. This framework is currently being extended into an agentic interface that facilitates the training and evaluation of diverse datasets, providing an accessible environment for users without programming experience.


Why Functions Don't Return Values
These functions are specifically architected to serve as tools for autonomous ML agents that will be used by other lab members for training and testing on their own datasets. In an agent-based workflow:


    

  1. State Management: All artifacts (trained models, generated plots, preprocessed data, evaluation metrics) are persisted to disk and managed through a directory-based state system rather than in-memory variables.
    

    2. 

  2. Agent Memory: The agent maintains its own state and memory by tracking file paths and metadata, not Python objects. This allows the agent to:
    

      

    • Resume workflows after interruptions
      • 

    • Share intermediate results across different execution contexts
      • 

    • Maintain a persistent record of all operations
      • 

    

    3. 

  3. Tool Composition: Each function acts as an independent, stateless tool that can be orchestrated by the agent. T"he agent decides when to call each tool based on the task requirements and maintains the overall workflow state.
    

    4. 

  4. Multi-User Scenarios: When multiple lab members deploy agents on different datasets, the directory-based approach ensures clear separation of artifacts and prevents memory conflicts.
    

    5. 



Why Functions Cannot Return Objects to Agents
Agent tools are fundamentally different from traditional Python functions.


When a human writes code interactively, they can capture return values:


model = train_model(data_path)
predictions = model.predict(X_test)



Agents cannot do this. When an agent calls a tool, the execution flow is:


    

  1. Agent invokes: train_model(data_path="/data/train.csv")
    2. 

  2. Function executes and returns a value
    3. 

  3. The returned value is immediately discarded - agents have no Python runtime to store variables
    4. 

  4. Agent receives only a text confirmation (e.g., "Model training completed")
    5. 



If a function returns a trained model object, that object vanishes the moment the function completes. The agent cannot assign it to a variable, cannot pass it to another function, and cannot inspect it. The return value is lost.


The Solution: Disk-Based State Management
Since agents cannot maintain in-memory state through variable assignment, all artifacts must be persisted to disk:


def train_model(data_path, model_dir):
    """Agent-compatible tool function"""
    model = RandomForestClassifier()
    model.fit(X, y)
    
    # Save to disk - this is the ONLY way the agent can access it later
    joblib.dump(model, f"{model_dir}/model.pkl")
    
    # Return value is just a status message for the agent's log
    return f"Model trained and saved to {model_dir}/model.pkl"



The agent's workflow now relies on the filesystem as its working memory:


Agent Internal State:


{
  "trained_model_path": "/models/run_2024/model.pkl",
  "preprocessed_data": "/data/processed/features.csv",
  "evaluation_results": "/results/metrics.json"
}



When the agent needs to evaluate the model, it calls:


evaluate_model(model_path="/models/run_2024/model.pkl", 
               test_data="/data/test.csv",
               output_dir="/results/")



The evaluation function loads the model from disk, processes it, and saves results back to disk. The agent never touches the Python objects directly, it only tracks file paths in its internal state. This helps with memory management, context window persistence and information retrieval when needed.


Key Architectural Principles
    

  1. Functions are stateless tools: Each function performs a discrete operation, reading inputs from disk and writing outputs to disk.
    

    2. 

  2. Agent maintains execution context: The agent's memory tracks file paths, execution history, and workflow state, not Python objects.
    

    3. 

  3. Persistence over performance: While loading/saving from disk is slower than in-memory operations, it enables workflow resumption, multi-session continuity, and multi-user isolation.
    

    4. 

  4. Directory structure as API: The organized directory layout (/data/, /models/, /results/) serves as the interface contract between tools.
    

    5.
